## 1. Что реально сломается на 100+ (оценка рисков)

| # | Риск | Где | Что происходит на 100+ | Серьёзность |
|---|------|-----|------------------------|-------------|
| R1 | Список профилей тянет всё, включая **аватары-base64** | `store.list()` → `/api/profiles`, `/api/admin/profiles`, лидеры, сообщество, бадди, дашборд, `recipients()` | Каждый вызов — мегабайты JSON (100 × ~30–100 КБ аватар). Медленно, дорого, риск таймаутов и лишнего трафика | 🔴 Critical |
| R2 | Массовая DM-рассылка **последовательная, без троттла и без учёта таймаута** | `notifications.ts deliver()` (цикл `for … await`), запускается в рамках HTTP-запроса | 100+ последовательных вызовов Telegram внутри serverless-функции (`maxDuration 30s`) → обрыв на середине, часть не доходит; нет retry на 429 | 🟠 High |
| R3 | Бэкофилл `@username` **блокирует загрузку списка** админки | `/api/admin/profiles` → `backfillUsernames()` | N `getChat`-вызовов на каждый холодный старт для тех, у кого username в TG нет; тормозит список, риск rate-limit Telegram | 🟡 Medium |
| R4 | Логин админки **без rate-limit**, один общий пароль в localStorage, CORS открыт | `/api/admin/login`, `x-admin-token` | Брутфорс пароля не ограничен; один пароль на всех админов; токен не истекает | 🟡 Medium (security) |
| R5 | Cron уведомлений **раз в день** (Vercel Hobby) | `vercel.json` cron `/api/cron/notify` | Событийные DM (эфир Чт 19:00 и т.п.) не попадают в нужный час | 🟡 Medium |
| R6 | Каталог достижений/ролей **в двух местах** | `server/src/content.ts` + `frontend/src/mock.ts` | Рассинхрон при правках → расхождение звёзд/названий | 🟢 Low |
| R7 | Нет **мониторинга ошибок, бэкапа-рутины и смоук-тестов** | — | На массовом онбординге инциденты незаметны, откат вслепую | 🟡 Medium (operability) |

## 2. План по фазам

### P0 — до раскатки (обязательно)
- [ ] **R1 — облегчить списки.** Отдавать в списочных эндпоинтах проекцию без тяжёлых полей (без `avatar`-base64): краткий формат для сообщества/лидеров/CRM-списка; полный профиль и аватар — по требованию (`/api/profile/:id`). Параллельно спланировать перенос аватаров в **Supabase Storage** (URL вместо base64) как целевое решение.
- [ ] **R2 — устойчивая рассылка.** Троттлинг (≤ ~25 msg/s), чанки, обработка `429 retry_after`, и вынос долгой отправки из HTTP-запроса (фоновая обработка/резюме по offset), чтобы не упираться в таймаут функции. Минимум — ограничить единичную рассылку размером и явно показывать прогресс/недоставленных.
- [ ] **R4 — базовая безопасность.** Rate-limit на `/api/admin/login` (N попыток/мин по IP), сузить CORS до доменов Mini App/админки, подтвердить `ALLOW_DEV_AUTH=0` в проде.
- [ ] **Бэкап + staged rollout.** Snapshot Supabase перед массовым онбордингом; раскатка волнами (см. плейбук §3).

### P1 — во время / сразу после раскатки
- [ ] **R3 — бэкофилл username.** Убрать из загрузки списка; оставить только в `/api/launch` (пишется при входе) + ручная кнопка «Обновить ссылки» в админке. Помечать «нет username» персистентно, чтобы не долбить `getChat`.
- [ ] **R5 — точные уведомления.** Внешний ежечасный пингер на `/api/cron/notify?secret=…` (обходит суточный лимит Hobby) или переход на Vercel Pro (hourly cron).
- [ ] **R7 — наблюдаемость.** Подключить трекинг ошибок (Sentry или структурные логи + алерты) и смоук-тесты критичных путей: register → launch → сохранение профиля → админ-логин → тестовая рассылка.

### P2 — усиление (после стабильной раскатки)
- [ ] **R6 — единый каталог.** Один источник достижений/ролей/уровней (общий модуль или из БД) вместо `content.ts` + `mock.ts`.
- [ ] Серверная пагинация/поиск для списков (когда список станет большим).
- [ ] Роли админов + истечение/ротация токена админки.

## 3. Плейбук раскатки на 100+

1. **Бэкап БД** — snapshot Supabase (точка отката данных).
2. **Смоук-прогон** критичных путей на проде (лучше на тестовых записях): регистрация → запуск → сохранение профиля → вход в админку → тестовая рассылка себе.
3. **Волнами:** 10 → 30 → 100 (по чатам/сегментам клуба). Между волнами — смотреть дашборд (запуски, ошибки), проверять доставку уведомлений.
4. **Точка отката:** предыдущий Vercel-деплой (`vercel rollback` / редеплой прошлого коммита) + восстановление БД из snapshot.

---

## 4. Метрики успеха харденинга

- p95 ответа `/api/admin/profiles` и `/api/profiles` — приемлемое время при 100 профилях (списки без аватаров-base64).
- Массовая рассылка на 100+ доходит **полностью**, без обрыва по таймауту и без потерь на 429.
- Онбординг волнами без инцидентов доступа/данных; есть бэкап и рабочий откат.
- Событийные уведомления приходят в правильный час.

---

## 5. Порядок работ (предлагаемый)

Начать с **R1 (облегчить списки)** — это и самый крупный риск, и разблокирует всё
остальное (рассылка, дашборд, админ-список станут быстрыми). Затем **R2 (рассылка)**
и **R4 (безопасность)** — и можно запускать первую волну.
